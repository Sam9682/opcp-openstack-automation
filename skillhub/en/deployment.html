Deployment Engine

Difficulty: Advanced · Estimated time: 30 min

The DeploymentEngine orchestrates the entire infrastructure lifecycle — networks, security groups, instances, and volumes — in the correct dependency order. It tracks every resource, rolls back on failure, and handles API rate limits with exponential backoff.

1. Full Orchestration

A single deploy_infrastructure() call creates all resources in dependency order: networks first, then security groups, then instances, and finally volumes attached to those instances. 

from openstack_sdk.deployment_engine import DeploymentEngine
from openstack_sdk.auth_manager import AuthenticationManager
from config.models import (
    DeploymentConfig, NetworkSpec, SubnetSpec,
    SecurityGroupSpec, SecurityGroupRule,
    InstanceSpec, VolumeSpec,
)

auth_mgr = AuthenticationManager()
credentials = auth_mgr.load_credentials_from_env()

config = DeploymentConfig(
    auth_url="${OS_AUTH_URL}",
    username="${OS_USERNAME}",
    password="${OS_PASSWORD}",
    tenant_name="${OS_TENANT_NAME}",
    region="${OS_REGION_NAME}",
    project_name="${OS_TENANT_NAME}",
    networks=[
        NetworkSpec(
            name="app-network",
            subnets=[
                SubnetSpec(
                    name="app-subnet",
                    cidr="192.168.1.0/24",
                    dns_nameservers=["8.8.8.8", "8.8.4.4"],
                )
            ],
        )
    ],
    security_groups=[
        SecurityGroupSpec(
            name="web-sg",
            description="Web traffic",
            rules=[
                SecurityGroupRule(
                    direction="ingress",
                    protocol="tcp",
                    port_range_min=80,
                    port_range_max=80,
                    remote_ip_prefix="0.0.0.0/0",
                    ethertype="IPv4",
                )
            ],
        )
    ],
    instances=[
        InstanceSpec(
            name="web-server",
            flavor="s1-2",
            image="Ubuntu 22.04",
            key_name="my-ssh-key",
            network_ids=[],
            security_groups=["web-sg"],
            metadata={"role": "web"},
        )
    ],
    volumes=[
        VolumeSpec(
            name="data-volume",
            size=100,
            volume_type="classic",
            attach_to="web-server",
        )
    ],
)

engine = DeploymentEngine(credentials)
result = engine.deploy_infrastructure(config)

if result.success:
    print(f"Deployment {result.deployment_id} succeeded")
    print(f"Duration: {result.duration_seconds:.1f}s")
    for rtype, ids in result.created_resources.items():
        print(f"  {rtype}: {len(ids)} resource(s)")
else:
    print(f"Deployment {result.deployment_id} failed")
    for f in result.failed_resources:
        print(f"  {f.resource_type}: {f.error_message}")

2. Deployment Tracking

Every deployment receives a unique deployment_id. The result object contains created resource IDs, failed resources, duration, and any orphaned resources that could not be cleaned up. 

# Save the deployment ID for later reference
with open("deployment_id.txt", "w") as f:
    f.write(result.deployment_id)

# Inspect created resources
print(result.created_resources)
# {'networks': ['net-abc'], 'instances': ['srv-123'], ...}

3. Automatic Rollback

If any resource fails to create, the engine automatically rolls back all previously created resources in reverse dependency order. This prevents orphaned infrastructure. 

from openstack_sdk.deployment_engine import DeploymentError

try:
    result = engine.deploy_infrastructure(config)

    if not result.success and result.orphaned_resources:
        print("Orphaned resources requiring manual cleanup:")
        for orphan in result.orphaned_resources:
            print(f"  - {orphan}")
except DeploymentError as exc:
    print(f"Deployment error: {exc}")

4. Manual Cleanup

You can also clean up specific resources by type and ID. 

resource_ids = {
    "instances": ["instance-id-1", "instance-id-2"],
    "volumes": ["volume-id-1"],
    "networks": ["network-id-1"],
}

results = engine.cleanup_resources(resource_ids)

for key, success in results.items():
    status = "ok" if success else "FAILED"
    print(f"  {key}: {status}")

5. Rate Limiting

The engine uses exponential backoff when the OpenStack API returns rate-limit errors (HTTP 429). This ensures deployments complete even under heavy API load without overwhelming the control plane. 

# Rate limiting is handled internally:
#   1st retry  → wait 1 s
#   2nd retry  → wait 2 s
#   3rd retry  → wait 4 s
#   …up to a configurable maximum

# Resource caching also reduces API calls:
# - Flavors, images, and networks are cached after first lookup
# - Subsequent lookups hit the cache instead of the API

Summary

  • The engine deploys resources in dependency order: networks → security groups → instances → volumes.
  • Every deployment gets a unique deployment_id for tracking.
  • On failure, automatic rollback removes previously created resources.
  • Use cleanup_resources() for manual teardown.
  • Exponential backoff handles API rate limits gracefully.
