Configuration

Difficulty: Beginner · Estimated time: 20 min

Every deployment starts with a configuration file that describes the infrastructure you want. The config/config_manager.py module loads YAML or JSON files, substitutes environment variables, and validates the result against strict data models.

1. YAML Configuration Format

YAML is the recommended format — it is human-readable and supports comments. Here is a complete example: 

# config.yaml — Full deployment configuration
auth_url: "${OS_AUTH_URL}"
auth_type: "${OS_AUTH_TYPE}"
region: "${OS_REGION_NAME}"
project_name: "${OS_TENANT_NAME}"
application_credential_id: "${OS_APPLICATION_CREDENTIAL_ID}"
application_credential_secret: "${OS_APPLICATION_CREDENTIAL_SECRET}"

networks:
  - name: "private-network"
    admin_state_up: true
    external: false
    subnets:
      - name: "private-subnet"
        cidr: "192.168.1.0/24"
        ip_version: 4
        enable_dhcp: true
        dns_nameservers:
          - "8.8.8.8"
          - "8.8.4.4"

security_groups:
  - name: "web-security-group"
    description: "Security group for web servers"
    rules:
      - direction: "ingress"
        protocol: "tcp"
        port_range_min: 80
        port_range_max: 80
        remote_ip_prefix: "0.0.0.0/0"
        ethertype: "IPv4"

instances:
  - name: "web-server-1"
    flavor: "s1-4"
    image: "Ubuntu 22.04"
    key_name: "my-ssh-key"
    network_ids:
      - "private-network"
    security_groups:
      - "web-security-group"

volumes:
  - name: "data-volume-1"
    size: 100
    volume_type: "classic"
    attach_to: "web-server-1"

2. JSON Configuration Format

The same configuration can be expressed in JSON. This is useful when generating configs programmatically. 

{
  "auth_url": "${OS_AUTH_URL}",
  "region": "${OS_REGION_NAME}",
  "project_name": "${OS_TENANT_NAME}",
  "application_credential_id": "${OS_APPLICATION_CREDENTIAL_ID}",
  "application_credential_secret": "${OS_APPLICATION_CREDENTIAL_SECRET}",
  "networks": [
    {
      "name": "private-network",
      "subnets": [
        {
          "name": "private-subnet",
          "cidr": "192.168.1.0/24"
        }
      ]
    }
  ],
  "instances": [
    {
      "name": "web-server-1",
      "flavor": "s1-4",
      "image": "Ubuntu 22.04",
      "key_name": "my-ssh-key",
      "network_ids": ["private-network"]
    }
  ]
}

3. Environment Variable Substitution

Any value wrapped in ${VAR_NAME} is replaced at load time with the matching environment variable. This keeps secrets out of version control. 

from config.config_manager import ConfigurationManager

manager = ConfigurationManager()

# ${OS_AUTH_URL} etc. are replaced with actual env var values
config = manager.load_config("examples/config.yaml")
print(f"Auth URL: {config.auth_url}")
print(f"Region:   {config.region}")
# Make sure the variables are exported before loading
export OS_AUTH_URL="https://auth.cloud.ovh.net/v3"
export OS_REGION_NAME="GRA7"
export OS_TENANT_NAME="my-project"

4. Configuration Validation

After loading, always validate. The validator checks required fields, CIDR notation, port ranges, duplicate names, and cross-references between resources. 

from config.config_manager import ConfigurationManager

manager = ConfigurationManager()
config = manager.load_config("examples/config.yaml")

validation = manager.validate_config(config)

if validation.is_valid:
    print("Configuration is valid!")
else:
    print("Validation errors:")
    for error in validation.errors:
        print(f"  - {error}")

Common validation checks include:

  • auth_url must be a valid HTTPS URL
  • CIDR notation must be well-formed (e.g. 192.168.1.0/24)
  • Port ranges: port_range_min <= port_range_max
  • Volume attach_to must reference an existing instance name
  • No duplicate resource names within the same type

5. Data Models

The configuration is parsed into typed Python dataclasses defined in config/models.py. Key models include: 

# Simplified view of the data models
from dataclasses import dataclass
from typing import List, Optional, Dict

@dataclass
class SubnetSpec:
    name: str
    cidr: str
    ip_version: int = 4
    enable_dhcp: bool = True
    gateway_ip: Optional[str] = None
    dns_nameservers: List[str] = None

@dataclass
class NetworkSpec:
    name: str
    admin_state_up: bool = True
    external: bool = False
    subnets: List[SubnetSpec] = None

@dataclass
class InstanceSpec:
    name: str
    flavor: str
    image: str
    key_name: str
    network_ids: List[str] = None
    security_groups: List[str] = None
    user_data: Optional[str] = None
    metadata: Dict[str, str] = None

@dataclass
class VolumeSpec:
    name: str
    size: int
    volume_type: str = "classic"
    bootable: bool = False
    image_id: Optional[str] = None
    attach_to: Optional[str] = None

Summary

  • Use YAML for hand-written configs, JSON for generated ones.
  • ${VAR_NAME} syntax keeps secrets in environment variables.
  • Always validate before deploying — it catches typos early.
  • Typed data models give you IDE auto-complete and clear documentation.
