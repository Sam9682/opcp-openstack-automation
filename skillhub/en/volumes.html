Volumes

Difficulty: Intermediate · Estimated time: 20 min

Block storage volumes provide persistent disk space that survives instance reboots and can be detached and reattached to different instances. This lesson covers creation, attachment, bootable volumes, and the exponential backoff polling strategy.

1. Creating Volumes

The VolumeManager creates volumes and waits for them to reach available status before returning. 

from openstack_sdk.auth_manager import ConnectionManager
from openstack_sdk.volume_manager import VolumeManager
from config.models import VolumeSpec, AuthCredentials

credentials = AuthCredentials(
    auth_url="${OS_AUTH_URL}",
    username="${OS_USERNAME}",
    password="${OS_PASSWORD}",
    tenant_name="${OS_TENANT_NAME}",
    region="${OS_REGION_NAME}",
)

conn_manager = ConnectionManager(credentials)
connection = conn_manager.connect()

volume_manager = VolumeManager(connection)

volume_specs = [
    VolumeSpec(name="data-vol-1", size=100, volume_type="classic"),
    VolumeSpec(name="fast-vol-1", size=50, volume_type="high-speed"),
]

volumes = volume_manager.create_and_attach_volumes(volume_specs, {})
for vol in volumes:
    print(f"{vol.name}  Size={vol.size}GB  Status={vol.status}")

2. Attaching Volumes to Instances

Pass an attach_to field referencing an instance name. The manager verifies the instance is ACTIVE before attaching. 

from openstack_sdk.compute_manager import ComputeManager

compute = ComputeManager(connection)
instances = compute.list_instances()
instances_dict = {inst.name: inst for inst in instances}

volume_specs = [
    VolumeSpec(
        name="web-data",
        size=150,
        volume_type="classic",
        attach_to="web-server-1",
    ),
]

volumes = volume_manager.create_and_attach_volumes(
    volume_specs, instances_dict
)
for vol in volumes:
    print(f"Attached {vol.name} — Status: {vol.status}")

3. Bootable Volumes

A bootable volume contains an OS image and can be used as the root disk for an instance. Set bootable=True and provide an image_id. 

boot_vol = VolumeSpec(
    name="boot-volume",
    size=50,
    volume_type="high-speed",
    bootable=True,
    image_id="image-id-123",  # Replace with actual image ID
)

volumes = volume_manager.create_and_attach_volumes([boot_vol], {})

4. Status Polling with Exponential Backoff

The volume manager uses exponential backoff when polling for status changes. The interval starts at 2 seconds and increases to a maximum of 10 seconds. This reduces API load during busy periods. 

# State transitions the manager tracks:
#   Creating   → Available    (timeout: 60 s)
#   Available  → Attaching    (initiated by attach call)
#   Attaching  → In-use       (verification: 30 s)

# Polling intervals (exponential backoff):
#   2s → 4s → 8s → 10s (capped)

5. Managing Volumes

# List all volumes
for vol in volume_manager.list_volumes():
    print(f"  {vol.name}  {vol.size}GB  {vol.status}")

# Detach a volume
volume_manager.detach_volume("instance-id", "volume-id")

# Delete a volume
volume_manager.delete_volume("volume-id")

6. Volume Types

OVHcloud offers two volume types:

  • classic — standard performance block storage, good for backups and archives.
  • high-speed — SSD-backed storage for databases and latency-sensitive workloads.
# In your config.yaml
volumes:
  - name: "data-volume"
    size: 100
    volume_type: "classic"
    attach_to: "web-server-1"

  - name: "db-volume"
    size: 50
    volume_type: "high-speed"
    attach_to: "db-server-1"

Summary

  • Volumes provide persistent block storage independent of instance lifecycle.
  • The manager verifies instances are ACTIVE before attaching.
  • Bootable volumes need bootable=True and a valid image_id.
  • Exponential backoff (2 s → 10 s) keeps API load reasonable during polling.
