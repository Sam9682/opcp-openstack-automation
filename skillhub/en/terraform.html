Terraform

Difficulty: Intermediate · Estimated time: 30 min

Terraform provides a declarative approach to infrastructure. You describe the desired state in HCL files and Terraform figures out the create/update/delete operations needed to reach it. This lesson walks through the project's Terraform configuration.

1. Provider Setup

The OpenStack provider connects Terraform to your OVHcloud project. Sensitive values come from variables (never hard-coded). 

provider "openstack" {
  auth_url                      = var.auth_url
  application_credential_id     = var.application_credential_id
  application_credential_secret = var.application_credential_secret
  tenant_name                   = var.tenant_name
  region                        = var.region
}
# Initialize the provider
cd terraform
terraform init

2. Network Resources

# Private network
resource "openstack_networking_network_v2" "private_network" {
  name           = var.network_name
  admin_state_up = true
}

# Subnet with CIDR validation
resource "openstack_networking_subnet_v2" "private_subnet" {
  name            = "${var.network_name}-subnet"
  network_id      = openstack_networking_network_v2.private_network.id
  cidr            = var.subnet_cidr
  ip_version      = 4
  dns_nameservers = var.dns_nameservers
  enable_dhcp     = var.enable_dhcp
}

3. Compute Instances

Use count to create multiple identical instances. Each instance depends on the subnet being ready. 

resource "openstack_compute_instance_v2" "instance" {
  count       = var.instance_count
  name        = "${var.network_name}-instance-${count.index + 1}"
  flavor_name = var.instance_flavor
  image_name  = var.instance_image
  key_pair    = var.key_name

  network {
    uuid = openstack_networking_network_v2.private_network.id
  }

  metadata  = var.instance_metadata
  user_data = var.instance_user_data

  depends_on = [
    openstack_networking_subnet_v2.private_subnet
  ]
}

4. Volumes and Attachments

# Block storage volumes
resource "openstack_blockstorage_volume_v3" "volume" {
  count       = var.volume_count
  name        = "${var.network_name}-volume-${count.index + 1}"
  size        = var.volume_size
  volume_type = var.volume_type
  image_id    = var.volume_bootable ? var.volume_image_id : null
}

# Attach volumes to instances (1:1 mapping)
resource "openstack_compute_volume_attach_v2" "volume_attachment" {
  count       = min(var.volume_count, var.instance_count)
  instance_id = openstack_compute_instance_v2.instance[count.index].id
  volume_id   = openstack_blockstorage_volume_v3.volume[count.index].id

  depends_on = [
    openstack_compute_instance_v2.instance,
    openstack_blockstorage_volume_v3.volume
  ]
}

5. Variables

Variables keep your configuration flexible. Use validation blocks to catch errors at plan time. 

variable "subnet_cidr" {
  description = "CIDR for the subnet"
  type        = string
  default     = "192.168.1.0/24"

  validation {
    condition     = can(cidrhost(var.subnet_cidr, 0))
    error_message = "Must be valid CIDR notation."
  }
}

variable "volume_size" {
  description = "Volume size in GB"
  type        = number
  default     = 100

  validation {
    condition     = var.volume_size > 0
    error_message = "Volume size must be positive."
  }
}

variable "instance_flavor" {
  description = "Instance flavor (e.g. s1-2, s1-4)"
  type        = string
  default     = "s1-2"
}

6. Outputs

Outputs expose key resource attributes after terraform apply. 

output "network_id" {
  description = "ID of the private network"
  value       = openstack_networking_network_v2.private_network.id
}

output "instance_ips" {
  description = "IP addresses of all instances"
  value       = openstack_compute_instance_v2.instance[*].access_ip_v4
}

output "deployment_summary" {
  value = {
    network   = openstack_networking_network_v2.private_network.name
    instances = openstack_compute_instance_v2.instance[*].name
    volumes   = openstack_blockstorage_volume_v3.volume[*].name
  }
}

7. State Management

Terraform stores the mapping between your config and real resources in a state file. For team use, store state remotely. 

# Plan — preview changes without applying
terraform plan -var-file="terraform.tfvars"

# Apply — create / update resources
terraform apply -auto-approve

# Destroy — tear down everything
terraform destroy -auto-approve

# Show current state
terraform show
# terraform.tfvars — variable values
auth_url                      = "${OS_AUTH_URL}"
application_credential_id     = "${OS_APPLICATION_CREDENTIAL_ID}"
application_credential_secret = "${OS_APPLICATION_CREDENTIAL_SECRET}"
tenant_name                   = "${OS_TENANT_NAME}"
region                        = "${OS_REGION_NAME}"
instance_image                = "Ubuntu 22.04"
key_name                      = "my-ssh-key"
instance_count                = 2
volume_count                  = 2

Summary

  • The OpenStack provider connects Terraform to OVHcloud.
  • Use variables with validation blocks for safe, reusable configs.
  • count enables scaling instances and volumes.
  • Outputs expose IPs, IDs, and summaries after apply.
  • Manage state carefully — use remote backends for teams.
