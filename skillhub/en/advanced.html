1. Multi-Region Deployments

OVHcloud offers multiple regions. Deploy the same stack to several regions for redundancy or latency optimization. 

from openstack_sdk.deployment_engine import DeploymentEngine
from openstack_sdk.auth_manager import AuthenticationManager

auth_mgr = AuthenticationManager()

regions = ["GRA7", "SBG5", "BHS5"]

for region in regions:
    credentials = auth_mgr.load_credentials_from_env()
    # Override region per iteration
    credentials.region = region

    engine = DeploymentEngine(credentials)
    result = engine.deploy_infrastructure(config)
    print(f"Region {region}: {'OK' if result.success else 'FAILED'}")

2. CI/CD Integration

Automate deployments with a CI/CD pipeline. The example below uses a generic pipeline definition — adapt it to GitHub Actions, GitLab CI, or Jenkins. 

# .ci/deploy.yaml — generic pipeline
stages:
  - validate
  - plan
  - deploy

validate:
  script:
    - python -c "
        from config import ConfigurationManager;
        m = ConfigurationManager();
        c = m.load_config('config.yaml');
        v = m.validate_config(c);
        assert v.is_valid, v.errors
      "

plan:
  script:
    - cd terraform
    - terraform init
    - terraform plan -out=tfplan

deploy:
  script:
    - cd terraform
    - terraform apply tfplan
  when: manual
# Environment variables for the pipeline
export OS_AUTH_URL="${OS_AUTH_URL}"
export OS_APPLICATION_CREDENTIAL_ID="${OS_APPLICATION_CREDENTIAL_ID}"
export OS_APPLICATION_CREDENTIAL_SECRET="${OS_APPLICATION_CREDENTIAL_SECRET}"
export OS_REGION_NAME="${OS_REGION_NAME}"
export OS_TENANT_NAME="${OS_TENANT_NAME}"

3. Custom Modules

Wrap common patterns into reusable Python functions or Terraform modules. 

"""custom_modules/web_stack.py — reusable web stack module."""

from config.models import (
    NetworkSpec, SubnetSpec, SecurityGroupSpec,
    SecurityGroupRule, InstanceSpec, VolumeSpec,
)


def web_stack(name, instance_count=2, volume_size=100):
    """Return a complete web-stack configuration."""
    return {
        "networks": [
            NetworkSpec(
                name=f"{name}-network",
                subnets=[
                    SubnetSpec(
                        name=f"{name}-subnet",
                        cidr="192.168.1.0/24",
                        dns_nameservers=["8.8.8.8"],
                    )
                ],
            )
        ],
        "security_groups": [
            SecurityGroupSpec(
                name=f"{name}-sg",
                description=f"SG for {name}",
                rules=[
                    SecurityGroupRule(
                        direction="ingress",
                        protocol="tcp",
                        port_range_min=80,
                        port_range_max=80,
                        remote_ip_prefix="0.0.0.0/0",
                        ethertype="IPv4",
                    ),
                ],
            )
        ],
        "instances": [
            InstanceSpec(
                name=f"{name}-web-{i+1}",
                flavor="s1-4",
                image="Ubuntu 22.04",
                key_name="my-ssh-key",
                network_ids=[f"{name}-network"],
                security_groups=[f"{name}-sg"],
            )
            for i in range(instance_count)
        ],
        "volumes": [
            VolumeSpec(
                name=f"{name}-vol-{i+1}",
                size=volume_size,
                volume_type="classic",
                attach_to=f"{name}-web-{i+1}",
            )
            for i in range(instance_count)
        ],
    }

4. Best Practices

  • Never hard-code credentials — use environment variables or application credentials.
  • Validate before deploying — catch CIDR, port, and reference errors early.
  • Use version control — track every config change in Git.
  • Tag resources — metadata makes cost allocation and cleanup easier.
  • Automate teardown — always have a cleanup path to avoid orphaned resources.
  • Monitor deployments — log deployment IDs and durations for observability.
  • Rotate credentials — use short-lived application credentials and rotate regularly.

5. SDK vs Terraform vs Ansible

Each approach has strengths. Choose based on your team's workflow.

Aspect OpenStack SDK Terraform Ansible
Language Python HCL YAML
Paradigm Imperative Declarative Declarative / Procedural
State management Manual Built-in state file Stateless (idempotent)
Rollback Custom (DeploymentEngine) terraform destroy Manual / playbook
Best for Complex logic, dynamic configs Reproducible infra, team workflows Config management, multi-step provisioning 
# Quick comparison — deploy the same stack three ways

# SDK
python examples/deploy_example.py

# Terraform
cd terraform && terraform apply -auto-approve

# Ansible (when playbooks are ready)
ansible-playbook ansible/playbook.yml -e @vars.yml

Summary

  • Deploy to multiple regions for redundancy with a simple loop or provider aliases.
  • Integrate with CI/CD — validate, plan, then apply with manual approval.
  • Build reusable modules to standardize common patterns.
  • Choose the right tool: SDK for flexibility, Terraform for state management, Ansible for configuration.
