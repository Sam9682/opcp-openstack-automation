1. Déploiements multi-régions

OVHcloud propose plusieurs régions. Déployez la même pile dans plusieurs régions pour la redondance ou l'optimisation de la latence. 

from openstack_sdk.deployment_engine import DeploymentEngine
from openstack_sdk.auth_manager import AuthenticationManager

auth_mgr = AuthenticationManager()

regions = ["GRA7", "SBG5", "BHS5"]

for region in regions:
    credentials = auth_mgr.load_credentials_from_env()
    # Remplacer la région à chaque itération
    credentials.region = region

    engine = DeploymentEngine(credentials)
    result = engine.deploy_infrastructure(config)
    print(f"Région {region} : {'OK' if result.success else 'ÉCHOUÉ'}")

2. Intégration CI/CD

Automatisez les déploiements avec un pipeline CI/CD. L'exemple ci-dessous utilise une définition de pipeline générique — adaptez-la à GitHub Actions, GitLab CI ou Jenkins. 

# .ci/deploy.yaml — pipeline générique
stages:
  - validate
  - plan
  - deploy

validate:
  script:
    - python -c "
        from config import ConfigurationManager;
        m = ConfigurationManager();
        c = m.load_config('config.yaml');
        v = m.validate_config(c);
        assert v.is_valid, v.errors
      "

plan:
  script:
    - cd terraform
    - terraform init
    - terraform plan -out=tfplan

deploy:
  script:
    - cd terraform
    - terraform apply tfplan
  when: manual
# Variables d'environnement pour le pipeline
export OS_AUTH_URL="${OS_AUTH_URL}"
export OS_APPLICATION_CREDENTIAL_ID="${OS_APPLICATION_CREDENTIAL_ID}"
export OS_APPLICATION_CREDENTIAL_SECRET="${OS_APPLICATION_CREDENTIAL_SECRET}"
export OS_REGION_NAME="${OS_REGION_NAME}"
export OS_TENANT_NAME="${OS_TENANT_NAME}"

3. Modules personnalisés

Encapsulez les patterns courants dans des fonctions Python réutilisables ou des modules Terraform. 

"""custom_modules/web_stack.py — module de pile web réutilisable."""

from config.models import (
    NetworkSpec, SubnetSpec, SecurityGroupSpec,
    SecurityGroupRule, InstanceSpec, VolumeSpec,
)


def web_stack(name, instance_count=2, volume_size=100):
    """Retourne une configuration complète de pile web."""
    return {
        "networks": [
            NetworkSpec(
                name=f"{name}-network",
                subnets=[
                    SubnetSpec(
                        name=f"{name}-subnet",
                        cidr="192.168.1.0/24",
                        dns_nameservers=["8.8.8.8"],
                    )
                ],
            )
        ],
        "security_groups": [
            SecurityGroupSpec(
                name=f"{name}-sg",
                description=f"GS pour {name}",
                rules=[
                    SecurityGroupRule(
                        direction="ingress",
                        protocol="tcp",
                        port_range_min=80,
                        port_range_max=80,
                        remote_ip_prefix="0.0.0.0/0",
                        ethertype="IPv4",
                    ),
                ],
            )
        ],
        "instances": [
            InstanceSpec(
                name=f"{name}-web-{i+1}",
                flavor="s1-4",
                image="Ubuntu 22.04",
                key_name="my-ssh-key",
                network_ids=[f"{name}-network"],
                security_groups=[f"{name}-sg"],
            )
            for i in range(instance_count)
        ],
        "volumes": [
            VolumeSpec(
                name=f"{name}-vol-{i+1}",
                size=volume_size,
                volume_type="classic",
                attach_to=f"{name}-web-{i+1}",
            )
            for i in range(instance_count)
        ],
    }

4. Bonnes pratiques

  • Ne jamais coder les identifiants en dur — utilisez des variables d'environnement ou des identifiants d'application.
  • Valider avant de déployer — détectez les erreurs de CIDR, ports et références tôt.
  • Utiliser le contrôle de version — suivez chaque changement de configuration dans Git.
  • Taguer les ressources — les métadonnées facilitent l'allocation des coûts et le nettoyage.
  • Automatiser le démontage — ayez toujours un chemin de nettoyage pour éviter les ressources orphelines.
  • Surveiller les déploiements — journalisez les IDs et durées de déploiement pour l'observabilité.
  • Rotation des identifiants — utilisez des identifiants d'application à durée limitée et effectuez une rotation régulière.

5. SDK vs Terraform vs Ansible

Chaque approche a ses forces. Choisissez en fonction du workflow de votre équipe.

Aspect OpenStack SDK Terraform Ansible
Langage Python HCL YAML
Paradigme Impératif Déclaratif Déclaratif / Procédural
Gestion de l'état Manuelle Fichier d'état intégré Sans état (idempotent)
Rollback Personnalisé (DeploymentEngine) terraform destroy Manuel / playbook
Idéal pour Logique complexe, configs dynamiques Infra reproductible, workflows d'équipe Gestion de config, provisionnement multi-étapes 
# Comparaison rapide — déployer la même pile de trois façons

# SDK
python examples/deploy_example.py

# Terraform
cd terraform && terraform apply -auto-approve

# Ansible (quand les playbooks sont prêts)
ansible-playbook ansible/playbook.yml -e @vars.yml

Résumé

  • Déployez dans plusieurs régions pour la redondance avec une simple boucle ou des alias de fournisseur.
  • Intégrez avec le CI/CD — validez, planifiez, puis appliquez avec approbation manuelle.
  • Construisez des modules réutilisables pour standardiser les patterns courants.
  • Choisissez le bon outil : SDK pour la flexibilité, Terraform pour la gestion d'état, Ansible pour la configuration.
