Terraform

Difficulté : Intermédiaire · Durée estimée : 30 min

Terraform offre une approche déclarative de l'infrastructure. Vous décrivez l'état souhaité dans des fichiers HCL et Terraform détermine les opérations de création/mise à jour/suppression nécessaires pour l'atteindre. Cette leçon parcourt la configuration Terraform du projet.

1. Configuration du fournisseur

Le fournisseur OpenStack connecte Terraform à votre projet OVHcloud. Les valeurs sensibles proviennent de variables (jamais codées en dur). 

provider "openstack" {
  auth_url                      = var.auth_url
  application_credential_id     = var.application_credential_id
  application_credential_secret = var.application_credential_secret
  tenant_name                   = var.tenant_name
  region                        = var.region
}
# Initialiser le fournisseur
cd terraform
terraform init

2. Ressources réseau

# Réseau privé
resource "openstack_networking_network_v2" "private_network" {
  name           = var.network_name
  admin_state_up = true
}

# Sous-réseau avec validation CIDR
resource "openstack_networking_subnet_v2" "private_subnet" {
  name            = "${var.network_name}-subnet"
  network_id      = openstack_networking_network_v2.private_network.id
  cidr            = var.subnet_cidr
  ip_version      = 4
  dns_nameservers = var.dns_nameservers
  enable_dhcp     = var.enable_dhcp
}

3. Instances de calcul

Utilisez count pour créer plusieurs instances identiques. Chaque instance dépend de la disponibilité du sous-réseau. 

resource "openstack_compute_instance_v2" "instance" {
  count       = var.instance_count
  name        = "${var.network_name}-instance-${count.index + 1}"
  flavor_name = var.instance_flavor
  image_name  = var.instance_image
  key_pair    = var.key_name

  network {
    uuid = openstack_networking_network_v2.private_network.id
  }

  metadata  = var.instance_metadata
  user_data = var.instance_user_data

  depends_on = [
    openstack_networking_subnet_v2.private_subnet
  ]
}

4. Volumes et attachements

# Volumes de stockage bloc
resource "openstack_blockstorage_volume_v3" "volume" {
  count       = var.volume_count
  name        = "${var.network_name}-volume-${count.index + 1}"
  size        = var.volume_size
  volume_type = var.volume_type
  image_id    = var.volume_bootable ? var.volume_image_id : null
}

# Attacher les volumes aux instances (correspondance 1:1)
resource "openstack_compute_volume_attach_v2" "volume_attachment" {
  count       = min(var.volume_count, var.instance_count)
  instance_id = openstack_compute_instance_v2.instance[count.index].id
  volume_id   = openstack_blockstorage_volume_v3.volume[count.index].id

  depends_on = [
    openstack_compute_instance_v2.instance,
    openstack_blockstorage_volume_v3.volume
  ]
}

5. Variables

Les variables rendent votre configuration flexible. Utilisez les blocs validation pour détecter les erreurs dès le plan. 

variable "subnet_cidr" {
  description = "CIDR du sous-réseau"
  type        = string
  default     = "192.168.1.0/24"

  validation {
    condition     = can(cidrhost(var.subnet_cidr, 0))
    error_message = "Doit être une notation CIDR valide."
  }
}

variable "volume_size" {
  description = "Taille du volume en Go"
  type        = number
  default     = 100

  validation {
    condition     = var.volume_size > 0
    error_message = "La taille du volume doit être positive."
  }
}

variable "instance_flavor" {
  description = "Flavor de l'instance (ex. s1-2, s1-4)"
  type        = string
  default     = "s1-2"
}

6. Sorties (Outputs)

Les sorties exposent les attributs clés des ressources après terraform apply. 

output "network_id" {
  description = "ID du réseau privé"
  value       = openstack_networking_network_v2.private_network.id
}

output "instance_ips" {
  description = "Adresses IP de toutes les instances"
  value       = openstack_compute_instance_v2.instance[*].access_ip_v4
}

output "deployment_summary" {
  value = {
    network   = openstack_networking_network_v2.private_network.name
    instances = openstack_compute_instance_v2.instance[*].name
    volumes   = openstack_blockstorage_volume_v3.volume[*].name
  }
}

7. Gestion de l'état

Terraform stocke la correspondance entre votre configuration et les ressources réelles dans un fichier d'état. Pour un usage en équipe, stockez l'état à distance. 

# Plan — prévisualiser les changements sans les appliquer
terraform plan -var-file="terraform.tfvars"

# Apply — créer / mettre à jour les ressources
terraform apply -auto-approve

# Destroy — tout supprimer
terraform destroy -auto-approve

# Afficher l'état actuel
terraform show
# terraform.tfvars — valeurs des variables
auth_url                      = "${OS_AUTH_URL}"
application_credential_id     = "${OS_APPLICATION_CREDENTIAL_ID}"
application_credential_secret = "${OS_APPLICATION_CREDENTIAL_SECRET}"
tenant_name                   = "${OS_TENANT_NAME}"
region                        = "${OS_REGION_NAME}"
instance_image                = "Ubuntu 22.04"
key_name                      = "my-ssh-key"
instance_count                = 2
volume_count                  = 2

Résumé

  • Le fournisseur OpenStack connecte Terraform à OVHcloud.
  • Utilisez des variables avec des blocs de validation pour des configurations sûres et réutilisables.
  • count permet la mise à l'échelle des instances et volumes.
  • Les sorties exposent les IPs, IDs et résumés après l'application.
  • Gérez l'état avec soin — utilisez des backends distants pour les équipes.
