Configuration

Difficulté : Débutant · Durée estimée : 20 min

Chaque déploiement commence par un fichier de configuration qui décrit l'infrastructure souhaitée. Le module config/config_manager.py charge des fichiers YAML ou JSON, substitue les variables d'environnement et valide le résultat par rapport à des modèles de données stricts.

1. Format de configuration YAML

YAML est le format recommandé — il est lisible par l'humain et prend en charge les commentaires. Voici un exemple complet : 

# config.yaml — Configuration complète de déploiement
auth_url: "${OS_AUTH_URL}"
auth_type: "${OS_AUTH_TYPE}"
region: "${OS_REGION_NAME}"
project_name: "${OS_TENANT_NAME}"
application_credential_id: "${OS_APPLICATION_CREDENTIAL_ID}"
application_credential_secret: "${OS_APPLICATION_CREDENTIAL_SECRET}"

networks:
  - name: "private-network"
    admin_state_up: true
    external: false
    subnets:
      - name: "private-subnet"
        cidr: "192.168.1.0/24"
        ip_version: 4
        enable_dhcp: true
        dns_nameservers:
          - "8.8.8.8"
          - "8.8.4.4"

security_groups:
  - name: "web-security-group"
    description: "Security group for web servers"
    rules:
      - direction: "ingress"
        protocol: "tcp"
        port_range_min: 80
        port_range_max: 80
        remote_ip_prefix: "0.0.0.0/0"
        ethertype: "IPv4"

instances:
  - name: "web-server-1"
    flavor: "s1-4"
    image: "Ubuntu 22.04"
    key_name: "my-ssh-key"
    network_ids:
      - "private-network"
    security_groups:
      - "web-security-group"

volumes:
  - name: "data-volume-1"
    size: 100
    volume_type: "classic"
    attach_to: "web-server-1"

2. Format de configuration JSON

La même configuration peut être exprimée en JSON. Cela est utile lorsque les configurations sont générées de manière programmatique. 

{
  "auth_url": "${OS_AUTH_URL}",
  "region": "${OS_REGION_NAME}",
  "project_name": "${OS_TENANT_NAME}",
  "application_credential_id": "${OS_APPLICATION_CREDENTIAL_ID}",
  "application_credential_secret": "${OS_APPLICATION_CREDENTIAL_SECRET}",
  "networks": [
    {
      "name": "private-network",
      "subnets": [
        {
          "name": "private-subnet",
          "cidr": "192.168.1.0/24"
        }
      ]
    }
  ],
  "instances": [
    {
      "name": "web-server-1",
      "flavor": "s1-4",
      "image": "Ubuntu 22.04",
      "key_name": "my-ssh-key",
      "network_ids": ["private-network"]
    }
  ]
}

3. Substitution des variables d'environnement

Toute valeur encadrée par ${VAR_NAME} est remplacée au chargement par la variable d'environnement correspondante. Cela permet de garder les secrets hors du contrôle de version. 

from config.config_manager import ConfigurationManager

manager = ConfigurationManager()

# Les ${OS_AUTH_URL} etc. sont remplacées par les valeurs réelles
config = manager.load_config("examples/config.yaml")
print(f"Auth URL: {config.auth_url}")
print(f"Region:   {config.region}")
# Assurez-vous que les variables sont exportées avant le chargement
export OS_AUTH_URL="https://auth.cloud.ovh.net/v3"
export OS_REGION_NAME="GRA7"
export OS_TENANT_NAME="my-project"

4. Validation de la configuration

Après le chargement, validez toujours. Le validateur vérifie les champs obligatoires, la notation CIDR, les plages de ports, les noms en double et les références croisées entre les ressources. 

from config.config_manager import ConfigurationManager

manager = ConfigurationManager()
config = manager.load_config("examples/config.yaml")

validation = manager.validate_config(config)

if validation.is_valid:
    print("La configuration est valide !")
else:
    print("Erreurs de validation :")
    for error in validation.errors:
        print(f"  - {error}")

Les vérifications courantes incluent :

  • auth_url doit être une URL HTTPS valide
  • La notation CIDR doit être bien formée (ex. 192.168.1.0/24)
  • Plages de ports : port_range_min <= port_range_max
  • Le champ attach_to d'un volume doit référencer un nom d'instance existant
  • Pas de noms de ressources en double au sein du même type

5. Modèles de données

La configuration est analysée en dataclasses Python typées définies dans config/models.py. Les principaux modèles incluent : 

# Vue simplifiée des modèles de données
from dataclasses import dataclass
from typing import List, Optional, Dict

@dataclass
class SubnetSpec:
    name: str
    cidr: str
    ip_version: int = 4
    enable_dhcp: bool = True
    gateway_ip: Optional[str] = None
    dns_nameservers: List[str] = None

@dataclass
class NetworkSpec:
    name: str
    admin_state_up: bool = True
    external: bool = False
    subnets: List[SubnetSpec] = None

@dataclass
class InstanceSpec:
    name: str
    flavor: str
    image: str
    key_name: str
    network_ids: List[str] = None
    security_groups: List[str] = None
    user_data: Optional[str] = None
    metadata: Dict[str, str] = None

@dataclass
class VolumeSpec:
    name: str
    size: int
    volume_type: str = "classic"
    bootable: bool = False
    image_id: Optional[str] = None
    attach_to: Optional[str] = None

Résumé

  • Utilisez YAML pour les configurations écrites à la main, JSON pour celles générées automatiquement.
  • La syntaxe ${VAR_NAME} garde les secrets dans les variables d'environnement.
  • Validez toujours avant de déployer — cela détecte les erreurs tôt.
  • Les modèles de données typés offrent l'auto-complétion dans l'IDE et une documentation claire.
