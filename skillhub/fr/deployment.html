Moteur de déploiement

Difficulté : Avancé · Durée estimée : 30 min

Le DeploymentEngine orchestre l'ensemble du cycle de vie de l'infrastructure — réseaux, groupes de sécurité, instances et volumes — dans le bon ordre de dépendances. Il suit chaque ressource, effectue un rollback en cas d'échec et gère les limites de débit API avec un backoff exponentiel.

1. Orchestration complète

Un seul appel à deploy_infrastructure() crée toutes les ressources dans l'ordre des dépendances : d'abord les réseaux, puis les groupes de sécurité, ensuite les instances, et enfin les volumes attachés à ces instances. 

from openstack_sdk.deployment_engine import DeploymentEngine
from openstack_sdk.auth_manager import AuthenticationManager
from config.models import (
    DeploymentConfig, NetworkSpec, SubnetSpec,
    SecurityGroupSpec, SecurityGroupRule,
    InstanceSpec, VolumeSpec,
)

auth_mgr = AuthenticationManager()
credentials = auth_mgr.load_credentials_from_env()

config = DeploymentConfig(
    auth_url="${OS_AUTH_URL}",
    username="${OS_USERNAME}",
    password="${OS_PASSWORD}",
    tenant_name="${OS_TENANT_NAME}",
    region="${OS_REGION_NAME}",
    project_name="${OS_TENANT_NAME}",
    networks=[
        NetworkSpec(
            name="app-network",
            subnets=[
                SubnetSpec(
                    name="app-subnet",
                    cidr="192.168.1.0/24",
                    dns_nameservers=["8.8.8.8", "8.8.4.4"],
                )
            ],
        )
    ],
    security_groups=[
        SecurityGroupSpec(
            name="web-sg",
            description="Trafic web",
            rules=[
                SecurityGroupRule(
                    direction="ingress",
                    protocol="tcp",
                    port_range_min=80,
                    port_range_max=80,
                    remote_ip_prefix="0.0.0.0/0",
                    ethertype="IPv4",
                )
            ],
        )
    ],
    instances=[
        InstanceSpec(
            name="web-server",
            flavor="s1-2",
            image="Ubuntu 22.04",
            key_name="my-ssh-key",
            network_ids=[],
            security_groups=["web-sg"],
            metadata={"role": "web"},
        )
    ],
    volumes=[
        VolumeSpec(
            name="data-volume",
            size=100,
            volume_type="classic",
            attach_to="web-server",
        )
    ],
)

engine = DeploymentEngine(credentials)
result = engine.deploy_infrastructure(config)

if result.success:
    print(f"Déploiement {result.deployment_id} réussi")
    print(f"Durée : {result.duration_seconds:.1f}s")
    for rtype, ids in result.created_resources.items():
        print(f"  {rtype} : {len(ids)} ressource(s)")
else:
    print(f"Déploiement {result.deployment_id} échoué")
    for f in result.failed_resources:
        print(f"  {f.resource_type} : {f.error_message}")

2. Suivi des déploiements

Chaque déploiement reçoit un deployment_id unique. L'objet résultat contient les IDs des ressources créées, les ressources en échec, la durée et les éventuelles ressources orphelines qui n'ont pas pu être nettoyées. 

# Sauvegarder l'ID de déploiement pour référence ultérieure
with open("deployment_id.txt", "w") as f:
    f.write(result.deployment_id)

# Inspecter les ressources créées
print(result.created_resources)
# {'networks': ['net-abc'], 'instances': ['srv-123'], ...}

3. Rollback automatique

Si la création d'une ressource échoue, le moteur effectue automatiquement un rollback de toutes les ressources précédemment créées dans l'ordre inverse des dépendances. Cela évite l'infrastructure orpheline. 

from openstack_sdk.deployment_engine import DeploymentError

try:
    result = engine.deploy_infrastructure(config)

    if not result.success and result.orphaned_resources:
        print("Ressources orphelines nécessitant un nettoyage manuel :")
        for orphan in result.orphaned_resources:
            print(f"  - {orphan}")
except DeploymentError as exc:
    print(f"Erreur de déploiement : {exc}")

4. Nettoyage manuel

Vous pouvez également nettoyer des ressources spécifiques par type et ID. 

resource_ids = {
    "instances": ["instance-id-1", "instance-id-2"],
    "volumes": ["volume-id-1"],
    "networks": ["network-id-1"],
}

results = engine.cleanup_resources(resource_ids)

for key, success in results.items():
    status = "ok" if success else "ÉCHOUÉ"
    print(f"  {key} : {status}")

5. Limitation de débit

Le moteur utilise un backoff exponentiel lorsque l'API OpenStack retourne des erreurs de limitation de débit (HTTP 429). Cela garantit que les déploiements aboutissent même sous forte charge API sans surcharger le plan de contrôle. 

# La limitation de débit est gérée en interne :
#   1ère tentative → attente 1 s
#   2ème tentative → attente 2 s
#   3ème tentative → attente 4 s
#   …jusqu'à un maximum configurable

# La mise en cache des ressources réduit aussi les appels API :
# - Flavors, images et réseaux sont mis en cache après la première recherche
# - Les recherches suivantes utilisent le cache au lieu de l'API

Résumé

  • Le moteur déploie les ressources dans l'ordre des dépendances : réseaux → groupes de sécurité → instances → volumes.
  • Chaque déploiement reçoit un deployment_id unique pour le suivi.
  • En cas d'échec, le rollback automatique supprime les ressources précédemment créées.
  • Utilisez cleanup_resources() pour un démontage manuel.
  • Le backoff exponentiel gère les limites de débit API de manière élégante.
